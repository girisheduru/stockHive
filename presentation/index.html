🐝 STOCKHIVE The Problem
Why we built it

Daily market signal — without the noise.

📰

Information overload

100 Nasdaq tickers × dozens of headlines × moving indicators = analyst burnout.

🎯

Single-prompt picks ≠ trustworthy

One chatbot prompt with "find me 5 stocks" hallucinates prices and sentiment.

⏰

Post-close decision window

Retail investors need a signal after the bell, ready before tomorrow's open.

We wanted to prove that an orchestrated team of small specialist agents, backed by deterministic scoring, beats a single mega-prompt — and ship it on a real schedule to a real channel.

🐝 STOCKHIVE Pipeline
The orchestrator's 6-stage contract

Strict order. Strict JSON. Zero hallucination.

Stage 1

Universe Selection

data-fetcher + pick_random10.py → 10 usable Nasdaq-100 tickers with return_4w.

Stage 2

Specialist Fan-Out

Technical, fundamental, and sentiment analysts dispatched in parallel from a single turn.

Stage 3

Aggregate

Orchestrator merges all three reports into one payload keyed by ticker.

Stage 4

Decide

decision_engine.py → market_view, top5, excluded, breadth metrics.

Stage 5

Publish

telegram-publisher renders canonical markdown and posts to channel.

Stage 6

Final Result

Single JSON run summary returned to the orchestrator session for continuity.

🐝 STOCKHIVE Agent Communication
How agents talk

Persistent orchestrator ↔ Ephemeral subagents

🎯 stockhive-orchestrator

PERSISTENT   Same session reused across every scheduled run.

  • Holds run-date, retry counters, continuity notes.
  • Issues structured JSON inputs to each subagent via the OpenClaw Task tool.
  • Validates each subagent's strict-JSON output before passing it forward.
  • Owns the only LLM-driven orchestration — never the scoring.

⚡ Specialist subagents (×5)

EPHEMERAL   Spawn → run → reply → tear down. No state bleed.

  • One-shot isolated context window per call.
  • Each binds to exactly one skill + scoped MCP tools.
  • Returns JSON only — guardrails forbid prose outside the contract.
  • Failure: retry once → fail loud → orchestrator stops the run.

Communication protocol

orchestrator
  └── Task(spawn: technical-analyst,
            input: [{ticker:"AVGO"}, {ticker:"AMD"}])
           ↓ strict-JSON reply
  └── orchestrator merges by ticker
  └── passes merged payload to decision_engine.py

Why ephemeral subagents?

  • Isolation — sentiment hallucination can't pollute the technical context.
  • Horizontal scale — 3 fan-out analysts run in one orchestrator turn.
  • Testable — each subagent runnable standalone with /agent run.
🐝 STOCKHIVE Subagents
5 specialists · one job each

Each subagent owns one stage of the run.

Subagent Stage Skill bound Output fields (per ticker) MCPs
data-fetcher 1 stock-data-fetcher ticker, return_4w, last_close yfinance nasdaq
technical-analyst 2a (parallel) technical-indicators rsi, macd_hist, sma20, sma50, close, signal, note alpha-vantage yfinance
fundamental-analyst 2b (parallel) fundamental-snapshot pe, market_cap, sector, earnings_health, note yfinance
sentiment-analyst 2c (parallel) sentiment-analyzer score [−1,+1], top_theme, n_headlines news-api

Each subagent is declared in agent-system/agents/<name>.md with frontmatter that lists its persistence, bound skill, and allow-listed tools. OpenClaw enforces those scopes at runtime.

🐝 STOCKHIVE MCP Servers
5 declarative external integrations

Every external API is an MCP server.

Wired declaratively through agent-system/mcps/mcp-config.json. OpenClaw spawns all five on /mcp connect. Subagents call mcp__<server>__<tool> within their scoped allow-list.

MCP ServerPurposeUsed byAuth
yfinance-mcp Yahoo Finance OHLCV + .info + .financials data-fetcher · technical-analyst · fundamental-analyst —
alpha-vantage-mcp RSI, MACD, SMA technical indicators technical-analyst ALPHA_VANTAGE_API_KEY
nasdaq-data-link-mcp Nasdaq Data Link historical EOD prices data-fetcher NASDAQ_DATA_LINK_API_KEY
news-api-mcp Recent ticker-tagged headlines for sentiment sentiment-analyst NEWS_API_KEY
telegram-bot-mcp Publishes markdown alerts to a Telegram channel telegram-publisher TELEGRAM_BOT_TOKEN + TELEGRAM_CHAT_ID
Why five? Yahoo Finance + Nasdaq Data Link cross-check prices. Alpha Vantage provides canonical indicator math. News API feeds an independent sentiment axis. Each subagent only sees the MCPs it needs — narrow surface area = fewer failure modes.
🐝 STOCKHIVE Ranking
From three signals to one ordered list

How the Top 5 are picked — no LLM in the scoring path.

composite  =  0.40·technical_score  +  0.30·fundamental_score  +  0.30·sentiment_score

📊 technical_score

  • BUY → +1.0 · HOLD → 0 · SELL → −1.0
  • Penalty: −0.5 if RSI > 70 (overbought)
  • Clamped to [−1, +1]
Source: alpha-vantage + yfinance

💰 fundamental_score

  • strong → +1.0 · mixed → +0.2 · weak → −0.6
  • Penalty: −0.5 if P/E > 80 (stretched valuation)
  • Clamped to [−1, +1]
Source: yfinance

📰 sentiment_score

  • Direct news score in [−1, +1]
  • Neutral fallback on thin coverage
  • Theme attached as rationale text
Source: news-api

Hard exclusion guardrails

  • RSI > 70 → EXCLUDED "overbought"
  • P/E > 80 → EXCLUDED "stretched"
  • Exclusions are reported in the final Telegram alert.

Market view rule

bullish = (
    breadth_buy_count >= 6  and
    sent_avg          >= 0.10 and
    rsi_avg           <  70
)
market_view = "BULLISH" if bullish else "BEARISH"
🐝 STOCKHIVE Determinism
The auditable core

Two Python scripts. Zero LLM. Fully reproducible.

pick_random10.py

Seeds random.Random with today's date string — the same trading day always shuffles to the same candidate order. Different day = different sample.

run_seed = seed or datetime.now().strftime("%Y-%m-%d")
rng = random.Random(run_seed)
shuffled = cleaned[:]
rng.shuffle(shuffled)
return shuffled[:n]
Same date → same 10 tickers across reruns, retries, and replays.

decision_engine.py

Pure-Python aggregator. Reads merged JSON on stdin, writes the decision JSON on stdout. Unit-testable with no API keys required.

  • Indexes 3 analyst reports by ticker.
  • Applies hard guardrail exclusions first.
  • Computes weighted composite for remaining tickers.
  • Sorts descending → emits Top 5 + market_view.
The scoring path never calls a model. Auditable, reproducible.
Design principle: let LLMs do what they're good at — reading messy financial prose and producing structured JSON — and let plain code do what it is good at: the math that must be right every time.
🐝 STOCKHIVE Scheduling
The cron job

One scheduled task. Weekday 17:00 ET, post-close.

nasdaq-daily-schedule.json

{
  "task_id":        "nasdaq-daily-top5-buys",
  "schedule":       "0 17 * * 1-5",
  "timezone":       "America/New_York",
  "agent":          "stockhive-orchestrator",
  "agent_session":  "persistent",
  "session_key":    "stockhive-orchestrator",
  "subagent_mode":  "ephemeral",
  "max_runtime_min": 20,
  "retry":          { "attempts": 2, "backoff_sec": 120 },
  "notify_on_failure": true
}

Behaviour

  • 📅 Cron 0 17 * * 1-5 — Mon–Fri at 17:00 (post-close).
  • 🔁 Binds back to the same persistent orchestrator session via session_key — not a fresh context per fire. The orchestrator preserves continuity.
  • ⚡ Subagents remain ephemeral per run — each run is isolated.
  • 🛡️ Retries twice with 120-second backoff before failure notification.
  • 👋 Manually triggerable any time: /task run nasdaq-daily-top5-buys
  • ⏸️ Pause/resume: /schedule disable / enable
Persistent session reuse is what makes this OpenClaw-native rather than just "a cron job that calls an LLM." The orchestrator keeps continuity — yesterday's notes inform today's run diagnostics.
🐝 STOCKHIVE Output
Where the alert lands

The hive ships honey to Telegram.

Publishing pipeline

  1. Decision JSON arrives at telegram-publisher subagent.
  2. Subagent invokes the telegram-formatter skill.
  3. Skill renders canonical markdown — date, market view, Top 5 with rationale, exclusions.
  4. mcp__telegram-bot-mcp__send_message publishes to the channel.
  5. Returns {telegram_message_id, chars_sent} as proof of delivery.

Setup in 3 steps

  • Create bot via @BotFather → /newbot
  • Add bot to channel as admin. Chat ID starts with -100…
  • Drop TELEGRAM_BOT_TOKEN + TELEGRAM_CHAT_ID into .env

🐝 STOCKHIVE Design Choices
The principles behind the architecture

Six decisions that shaped the hive.

1. Agent system, not a package

StockHive is a runtime composition, not an installable library. One manifest, one runtime — exactly what OpenClaw is built for.

2. Ephemeral by default

Each analyst spins up isolated, does its job, and is torn down on reply. No state bleed. Cheap to scale horizontally.

3. Parallel where possible

The serial bottleneck is only stage 1. Stage 2's three analysts dispatch in a single orchestrator turn — no waiting between them.

4. Deterministic decision layer

The BULLISH/BEARISH call and Top 5 ranking are reproducible, auditable, and unit-testable — independent of any model run.

5. Composable skills

Every skill is independently triggerable. Other agent systems in the workspace can reuse technical-indicators tomorrow.

6. One-command lifecycle

/agents register → /task run → /schedule enable. Teardown is a mirror command. No bespoke launchers.
